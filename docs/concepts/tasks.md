# Tasks

A task is the primary unit of work in DashR. Every agent invocation, every OpenClaw session, and every log entry belongs to a task. Tasks are persistent, trackable, and structured — they survive browser reloads, persist across server restarts, and carry a full audit trail from creation to completion.

---

## Task Anatomy

Every task has the following fields:

| Field | Type | Description |
|---|---|---|
| `id` | `string (uuid)` | Unique identifier for the task. Generated by DashR at creation time. |
| `title` | `string` | A short, human-readable label for the task. Used in Kanban cards and list views. |
| `prompt` | `string` | The full instruction sent to the agent when the session is invoked. |
| `agent_id` | `string` | The ID of the assigned agent (e.g., `connie`, `maxwell`). Set at creation or by Connie's router. |
| `owner` | `string` | The user or system that created the task. Used for attribution in multi-user contexts. |
| `status` | `enum` | Current lifecycle state. One of: `created`, `assigned`, `in-progress`, `waiting`, `blocked`, `completed`. |
| `priority` | `enum` | Task urgency. One of: `critical`, `high`, `medium`, `low`. |
| `model_id` | `string` | The OpenClaw model ID used for the session (e.g., `claude-sonnet-4-5`). Inherited from the agent's default model at assignment time. |
| `tokens_used` | `integer` | Total tokens consumed by the task's OpenClaw session. Updated incrementally as the session streams. |
| `cost_usd` | `float` | Estimated cost in USD. Calculated from `tokens_used` and the model's per-token pricing. |
| `openclaw_session_id` | `string` | The session ID returned by OpenClaw when the session is opened. Used to correlate logs and hook events. |
| `wbs` | `array` | The Work Breakdown Structure — an array of up to 4 subtask objects, each with its own status. |
| `logs` | `array` | Structured log entries attached to this task. Each log has a timestamp, level, and content. |

## Task Lifecycle

Tasks move through a defined set of states. Not all transitions are automatic — some require explicit action from the user or a hook event from OpenClaw.

```
                    ┌──────────────┐
                    │   created    │  ← Task is saved to the database
                    └──────┬───────┘
                           │ agent assigned (auto or manual)
                           ▼
                    ┌──────────────┐
                    │   assigned   │  ← Agent is selected; session not yet started
                    └──────┬───────┘
                           │ session opened via OpenClaw
                           ▼
                    ┌──────────────┐
              ┌────▶│ in-progress  │  ← Agent is actively working; logs streaming
              │     └──────┬───────┘
              │            │
              │     ┌──────┴───────┐
              │     │              │
              │     ▼              ▼
              │  ┌──────────┐  ┌──────────┐
              │  │ waiting  │  │ blocked  │  ← Waiting: paused pending input
              │  └────┬─────┘  └────┬─────┘     Blocked: cannot proceed
              │       │             │
              └───────┘             │ manually unblocked
                                    ▼
                    ┌──────────────────────┐
                    │      in-progress     │
                    └──────────┬───────────┘
                               │ session completes successfully
                               ▼
                    ┌──────────────────────┐
                    │      completed       │  ← Final state. Read-only.
                    └──────────────────────┘
```

**State descriptions:**

| State | Description |
|---|---|
| `created` | The task record exists in the database. No agent has been assigned yet, or the assignment is pending. |
| `assigned` | An agent has been selected. DashR is about to open an OpenClaw session. |
| `in-progress` | An OpenClaw session is open and the agent is actively processing the prompt. Logs are streaming. |
| `waiting` | The session is paused, typically because the agent is waiting for additional input or an external resource. The session remains open. |
| `blocked` | The task cannot proceed due to a dependency, conflict, or error. The session may be closed. Manual intervention is required to resume. |
| `completed` | The session has ended successfully. The task is read-only. All metrics are finalized. |

## Work Breakdown Structure (WBS)

When a task is created, DashR automatically generates a WBS — a 4-item checklist that breaks the task's prompt into discrete, trackable subtasks.

The WBS is generated by analyzing the prompt and producing four logical steps. For example, a task with the prompt "Write a competitive analysis of the top 3 CRM tools" might produce:

```
[x] 1. Identify the three CRM tools to compare
[ ] 2. Research key features and pricing for each
[ ] 3. Draft comparison across evaluation criteria
[ ] 4. Write executive summary and recommendation
```

Each WBS item has its own status (`pending`, `in-progress`, `done`) and can be updated independently via the UI or the `PATCH /api/tasks/:id/wbs/:wbs_id` endpoint. WBS items can also be marked complete by the agent itself if it emits structured progress events through the OpenClaw hook.

WBS progress is displayed on the task card in the Kanban view as a progress bar (e.g., `2 / 4`), and in full detail in the task panel.

## Task Priorities

| Priority | Color | Meaning |
|---|---|---|
| `critical` | Red | Must be addressed immediately. Appears at the top of all lists and boards. |
| `high` | Orange | Important and time-sensitive. Sorted above medium and low tasks. |
| `medium` | Yellow | Standard priority. The default for new tasks. |
| `low` | Gray | Background work. Agents will pick these up when higher-priority tasks are clear. |

Priority affects display order in list views and Kanban columns, but does not currently affect OpenClaw session scheduling. If you need strict execution ordering, create tasks sequentially rather than in bulk.

## How Tasks Connect to OpenClaw Sessions

Every task that reaches `in-progress` state has a corresponding OpenClaw session. DashR stores the session identifier returned by OpenClaw in the `openclaw_session_id` field.

This ID is used to:

- **Correlate hook events** — when OpenClaw fires a `session.progress` or `session.complete` hook, DashR matches the payload's session ID to a task and updates it accordingly.
- **Stream logs** — the WebSocket gateway uses the session ID to subscribe to the correct output stream.
- **Resume sessions** — if a task is unblocked, DashR can pass the original `openclaw_session_id` back to OpenClaw to continue the session rather than starting a new one (when supported by the model).

If `openclaw_session_id` is `null`, the task has not yet been submitted to OpenClaw.

## Creating Tasks

**Via the UI:**
Use the **New Task** button in the Kanban board header. Fill in the title, prompt, agent, and priority. DashR handles session creation automatically.

**Via the API:**
Send a `POST` request to `/api/tasks` with a JSON body:

```json
{
  "title": "Summarize the Q1 board report",
  "prompt": "Read the attached Q1 report and produce a 3-paragraph executive summary highlighting revenue, risk, and next steps.",
  "agent_id": "ada",
  "priority": "high",
  "owner": "tomas"
}
```

See the [API Overview](../api/overview.md) for full request and response documentation.

## Token Tracking and Cost Estimation

DashR tracks token usage at the task level by consuming the `usage` field from OpenClaw's session events. The `tokens_used` field is updated incrementally during streaming — you see the counter rise in the task panel as the agent works.

Cost is estimated using a per-model pricing table stored in the `models` table. The formula is:

```
cost_usd = (tokens_used / 1,000,000) * model.cost_per_million_tokens
```

This is an estimate. It uses input+output tokens combined against a blended rate. For precise billing, refer to your Anthropic account dashboard.

Cost and token data roll up to the agent level, giving you per-agent spending totals across all tasks — visible on the Agents page and via `GET /api/agents/:id`.
